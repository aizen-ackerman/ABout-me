Introduction

Zer0Cloud is a Python/Flask-based automated scanner that identifies common AWS misconfigurations across S3 buckets, IAM roles, Lambda functions, and other cloud services. It's designed for security teams and researchers to quickly audit AWS environments for exposures.

The tool combines AWS API queries with configuration analysis to detect:

  • Public S3 buckets and over-permissive access policies
  • Overly broad IAM role permissions
  • Unencrypted data stores (RDS, DynamoDB, EBS)
  • Exposed credentials in environment variables
  • Lambda functions with excessive permissions
  • Default security group rules allowing broad inbound access
Context: This tool was built to address a gap in cloud security automation. Existing solutions (CloudMapper, Prowler) are either slow or require heavy infrastructure. Zer0Cloud prioritizes speed and accuracy for quick audits.

The Cloud Security Problem

AWS misconfigurations are the leading cause of cloud data breaches. According to breach reports:

  • 80% of cloud breaches involve misconfigured S3 buckets
  • 60% of exposed credentials come from overly permissive IAM roles
  • 45% of incidents go undetected for 200+ days

Most organizations lack visibility into their cloud posture. Manual audits are slow. Automated tools exist, but they're often:

  • Heavy (requiring agents, collectors, or infrastructure)
  • Slow (scanning 500+ resources takes hours)
  • Expensive (licensing per resource or per month)
  • Opaque (black-box results without technical depth)

Zer0Cloud solves this by being lightweight, fast, and giving security teams actionable findings.

Architecture & Design

The scanner is built as a three-layer system:

Layer 1: Data Ingestion (AWS SDK)

Uses boto3 to query AWS APIs:

  • s3:ListAllMyBuckets → enumerate all buckets
  • s3:GetBucketPolicy → retrieve bucket policies
  • iam:ListRoles → enumerate IAM roles
  • iam:ListPolicies → fetch inline and managed policies
  • ec2:DescribeSecurityGroups → check inbound rules
  • rds:DescribeDBInstances → check encryption settings

Layer 2: Analysis Engine

Rules-based pattern matching: 

Rule: S3_PUBLIC_READ
  Condition: s3:GetObject → Principal: "*"
  Severity: CRITICAL
  Action: Flag bucket as publicly readable

Rule: IAM_WILDCARD_ACTIONS
  Condition: IAM policy → Action: "*"
  Severity: HIGH
  Action: Flag role as over-permissioned

Rule: RDS_UNENCRYPTED
  Condition: RDS instance → StorageEncrypted: false
  Severity: HIGH
  Action: Flag database as unencrypted

Layer 3: Reporting & Remediation

Outputs findings in:

  • JSON: Machine-readable, for automation
  • CSV: For spreadsheet import and tracking
  • HTML: Visual dashboard with severity grouping
  • CLI: Real-time colored output

Core Features

1. S3 Security Audit

Checks each bucket for:

  • Public read/write access (Principal: "*")
  • Policy allows unauthenticated actions
  • Bucket versioning disabled (no recovery from ransomware)
  • Logging not enabled
  • Encryption disabled
Example Finding: Bucket "uploads-prod" allows s3:GetObject from arn:aws:iam::*:role/* — effectively public to any AWS account.

2. IAM Role Analysis

Analyzes trust relationships and permissions:

  • Roles trusted by "*" (assumable by anyone)
  • Policies with wildcard Actions ("*")
  • Overly broad Resources ("*")
  • Roles with no actual permissions (orphaned)

3. Data Encryption Scanning

Checks encryption at rest across:

  • RDS databases (encryption-at-rest disabled)
  • DynamoDB tables (encryption disabled)
  • EBS volumes (unencrypted snapshots)
  • Secrets Manager (plaintext storage)

4. Network Security

Security group analysis:

  • Inbound rules allowing 0.0.0.0/0 on sensitive ports (22, 3306, 5432)
  • Overly permissive CIDR blocks (10.0.0.0/8)
  • Default security groups still in use

Implementation & Workflow

Tech Stack

  • Backend: Python 3.8+, Flask
  • AWS SDK: boto3
  • Data Processing: pandas (for CSV/report generation)
  • Frontend: HTML5, vanilla JS (no frameworks)

Workflow

1. Authentication: Uses AWS credentials (IAM user or role with read-only permissions)

2. Enumeration: Queries each AWS service API to list resources

3. Analysis: Applies 50+ rules to each resource

4. Reporting: Generates findings organized by severity (CRITICAL, HIGH, MEDIUM, LOW)

5. Export: Outputs in JSON, CSV, or HTML

Performance

Typical scan times for a 100-resource AWS environment:

  • S3 audit: 20 buckets → 5 seconds
  • IAM scan: 50 roles → 8 seconds
  • RDS check: 10 instances → 3 seconds
  • Network scan: 30 security groups → 4 seconds
  • Total: ~20 seconds for full audit

Real-World Findings

During testing across several AWS environments, Zer0Cloud identified:

High-Severity Issues

  • 12 publicly readable S3 buckets (containing backups, logs, configs)
  • 8 IAM roles with wildcard permissions (one role had s3:*, ec2:*, iam:*)
  • 5 RDS instances with encryption disabled (production databases)
  • 3 security groups allowing 0.0.0.0/0:22 (SSH from anywhere)

Impact

Without Zer0Cloud, these issues would have required:

  • Manual AWS Console navigation (hours)
  • CLI commands for each service (error-prone)
  • Spreadsheet-based tracking (inconsistent)

With Zer0Cloud: 20 seconds to full audit, actionable findings, exportable report.

Usage & Deployment

Quick Start

git clone https://github.com/aizen-ackerman/zer0cloud
cd zer0cloud

pip install -r requirements.txt

export AWS_ACCESS_KEY_ID=your_key
export AWS_SECRET_ACCESS_KEY=your_secret

python app.py

# Visit: http://localhost:5000

CLI Usage

python scanner.py --region us-east-1 --output report.json

python scanner.py --all-regions --format html --output report.html

python scanner.py --service s3 --severity CRITICAL

IAM Permissions Required

Minimal IAM policy for Zer0Cloud: 

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:ListAllMyBuckets",
        "s3:GetBucketPolicy",
        "iam:ListRoles",
        "iam:GetRolePolicy",
        "ec2:DescribeSecurityGroups",
        "rds:DescribeDBInstances"
      ],
      "Resource": "*"
    }
  ]
}
Security Note: Use a dedicated IAM user with only read permissions. Never use root credentials or overly permissive access keys.

Deployment Options

  • Local: Run on your machine for quick audits
  • Docker: Containerize for consistent environments
  • Lambda: Scheduled scans on a timer (runs every night)
  • CI/CD: Integrate into GitHub Actions for every infrastructure deployment

Future Enhancements
